# 調査結果サマリー（ブラックボックス）

## 1. 再現した事象（テストアカウント student ログイン時）

| 手順 | 結果 |
|------|------|
| `POST /auth/login`（`login_mode=student`、提供のユーザー名・パスワード） | `303` → `Location: /student/diary/` |
| `GET /auth/status`（セッション付き） | `200`、`role` は **`student_files`**、`user_id`・`username` が JSON で返る |
| `GET /student/diary/`（同一セッション） | **`403`**、本文に「**このページはスタッフ専用です。**」 |

**結論（原因の強い仮説）**: URL パスは `/student/...` だが、サーバ側の認可が **スタッフ用ガード** と誤結線されているか、**ロール名の不一致**（例: コードが `staff` のみ許可し、`student_files` を学生として扱っていない）で **生徒を拒否**している。

これは利用者が報告した「スタッフなのに弾かれる」とは別パターンだが、**認可ロジックの一貫性欠如**の証拠になる。スタッフ用ページと学生用ページで **別々のロール判定** がコピペずれしている可能性が高い。

## 2. セッション・クッキー（設計レビュー観点）

- **`portal_auth`**: 値が **パイプ区切りの複数フィールド**（先頭は数値バージョン、続けてロールらしき文字列、数値 ID、ユーザー名、時刻、末尾に固定長の 16 進列＝署名の可能性）で構成されている。**HttpOnly** および **SameSite=strict** が付与されている例を確認。
- **`Set-Cookie` に `Secure` 属性が見当たらない**（HSTS ありでも、仕様上は `Secure` 付与を推奨）。
- **`csrf_token` クッキー**: **HttpOnly なし**（JavaScript から読める）。XSS があると CSRF 対策が弱まり得るため、CSP と出力エスケープの確認を推奨。

※本レポートに **生のクッキー値は記載しない**。

## 3. HTTP ヘッダの重複

複数エンドポイントで **`Content-Security-Policy` が同一レスポンスに 2 本**出ている（アプリとリバースプロキシの二重付与の典型）。**1 経路に統一**することを推奨。

## 4. ログイン URL と HEAD

`/auth/login` に **HEAD** を送ると **405**（`Allow: GET`）となる挙動があり得る。一部クライアントやヘルスチェックでノイズになるため、**HEAD を許可**するか、意図を文書化する。

## 5. 「invalid response」について

本環境の `curl`（GET・フォロー）では `/student/diary/` は未ログイン時 **302**、ログイン後 **403 HTML** で、**不正な HTTP としての切断は再現していない**。ブラウザのみで出る場合は **HTTP/3（QUIC）** や **zstd 圧縮**、**PWA/Service Worker** を切り分け対象とする（前回メッセージのチェックリスト参照）。

## 6. 不正昇格・パスワード書き換え（本調査の範囲外）

- **クッキー改ざん・署名回避・他アカウントのパスワード API への操作は未実施**。
- 推奨する **社内での確認項目**（チェックリストのみ）:
  - すべての変更系 API で **ロールをサーバ側で再取得**し、クッキー内の平文フィールドを信頼しない。
  - パスワード変更エンドポイントで **対象ユーザ ID をクライアント任せにしない**（セッションの主体と一致検証、スタッフ操作は別途監査ログ）。
  - IDOR: `user_id` をパラメータで受け取る操作で **他ユーザの ID が指定できないか**。

## 7. 次のアクション（開発側）

1. **`/student/diary/` の認可条件**を仕様どおりに修正（`student_files` または `student` を許可するか、ルートを `/staff/...` に移すかを決定）。
2. **スタッフが弾かれる**報告について、スタッフログイン時の `role` 文字列と当該ガードの期待値をログで突き合わせる。
3. CSP 二重・`Secure` フラグ・`csrf_token` の HttpOnly を **セキュリティハードニング** タスクとして整理。
